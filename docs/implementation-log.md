# 実装履歴

## 2026-07-21 Cloudflare Web Analyticsの導入

### 目的

ゲームサイトへの訪問数、流入元、スマートフォン/PC比率を把握し、宣伝施策の効果を比較できるようにする。

### 実装内容

Cloudflareダッシュボードに `chika-hyakkei.github.io` のWeb Analyticsサイトを作成した。`app/layout.tsx` でCloudflare提供のbeaconを読み込み、フッターに匿名アクセス集計の利用を表示した。設計、送信範囲、制約は `docs/architecture.md`、`docs/data-and-save.md`、ADR-004へ記録した。

### 主要な処理の流れ

ページを開くとCloudflareのbeaconが匿名のページ訪問情報を集計する。集計結果はCloudflareダッシュボードの **Analytics → Web analytics → chika-hyakkei.github.io** で確認する。ゲームのセーブ、探索行動、名前、テスト記録はこの計測へ渡さない。

### 設計上の注意点

beaconトークンは公開ページに含まれる識別子であり管理権限を持たない。一方、Cloudflareの管理APIトークンやログイン情報はリポジトリに保存しない。追跡防止機能による未計測があり得るため、数字は施策の比較に使い、厳密な実人数とみなさない。

### 影響範囲

初回ページ表示時の外部通信、タイトル・ゲーム画面下部の案内、公開ビルド、開発資料。通常セーブ、テスト記録、ランキングデータ形式は変更しない。

### 未対応・今後の課題

「開始」「10階到達」などゲーム内の到達率まで測る場合は、別途、送信項目を限定したイベント計測を設計する。

## 2026-07-21 ヘッダータイトルのロゴ化

### 目的

タイトル画面の左上を、文字組みではなく地下百景専用の看板ロゴとして見せる。洞窟内の冒険イラストとは役割を分け、イラストの上へ重ねない。

### 実装内容

`public/assets/title-logo.webp` を追加し、`app/page.tsx` のヘッダーにあった英字キッカーと `h1` を `site-logo` の画像へ置き換えた。`app/globals.css` にはPC・スマートフォンそれぞれで記録表示と共存できる表示幅を追加した。元画像の背景は、外周につながる色だけを透明化して宝石装飾の色を残している。公開用は透過対応のWebPにして容量も抑える。

`app/updates.ts` にも今回の更新を追加し、タイトル画面の更新要約と履歴画面から確認できるようにした。

### 主要な処理の流れ

タイトル画面描画時に、ヘッダー左側が `/assets/title-logo.webp` を表示する。右側の最高記録、投票ボタン、音声ボタンは既存のまま表示する。タイトル下の洞窟イラストは独立しており、ロゴは重ねない。

### 設計上の注意点

ロゴの正本は公開用の `public/assets/title-logo.webp` とする。制作途中の画像は `promo/` に置き、追跡対象にしない。ヘッダー幅は狭い画面で操作ボタンを押し出さないようCSSで縮小する。

### 影響範囲

タイトル画面の見た目、公開アセット、更新履歴、静的レンダリングテスト。ゲーム進行とセーブデータ形式には影響しない。

### 未対応・今後の課題

実機でタイトルヘッダーの幅を確認し、必要ならロゴ表示幅だけを微調整する。

## 2026-07-20 タイトルの更新要約・週間TOP3と商店の満杯表示

### 目的

公開後の更新内容と今週の上位記録を、ゲーム開始前に邪魔にならない範囲で見せる。あわせて、持ち物が満杯の商店購入を誤解なく扱う。

### 実装内容

`app/updates.ts` を更新履歴のデータ元として追加した。タイトルの左下に最新更新と週間TOP3を表示し、更新履歴は別モーダルで全件を確認できるようにした。週間TOP3は既存ランキングAPIから読取専用で取得する。

商店購入も確認した。既存コードは、交換対象の装備があり持ち物8個なら金貨を減らす前に購入を中断していた。今回、該当商品のボタンを無効化して「持ち物がいっぱい」と表示し、UI上でも購入不可を明示した。

### 主要な処理の流れ

タイトル表示時に週間ランキングを最大3件だけ取得する。更新内容は静的な `UPDATE_NOTES` から最新1件を表示し、履歴ボタンで全件を表示する。ランキング取得に失敗した場合はTOP3を空表示にし、ゲーム開始を妨げない。

### 設計上の注意点

週間TOP3の読取は自動だが、プレイ結果の送信は任意のまま維持する。今後の更新時は `app/updates.ts` の先頭へ1件追加し、必要に応じてこの履歴も更新する。

### 影響範囲

タイトル画面、ランキングAPIの読取、商店UI、開発資料。セーブデータ形式は変更しない。

### 未対応・今後の課題

ランキングが空の場合の文言は表示するが、通信エラーの詳細はタイトル画面へ出さない。

## 2026-07-19 オンラインランキング基盤

### 目的

プレイヤーが任意で到達記録を共有できるようにする。

### 実装内容

ゲームに地下ランキング画面と結果送信UIを追加し、`ranking-api/` にCloudflare Worker/D1のAPIと移行SQLを追加した。ゲームはGitHub Pagesのまま、ランキングだけ外部APIに分離した。

### 主要な処理の流れ

結果画面で公開名を入力して送信すると、匿名端末ID単位の自己ベストがWorkerへ送られる。WorkerはD1の総合・週間ベストを更新し、ランキング画面は上位100件を取得する。

### 設計上の注意点

テストモードと途中放棄は送信しない。送信は任意で、自動送信しない。クライアント値を信頼するカジュアルランキングである。

### 影響範囲

`app/page.tsx`、`app/ranking.ts`、`public/ranking-config.js`、`ranking-api/`、Cloudflare D1/Worker、GitHub Pages。

### 未対応・今後の課題

不正対策を強化するなら、サーバー発行シードと行動ログ検証を設計する。

## 2026-07-21 終了記録の自動ランキングと今回行の強調

### 目的

死亡・中断を含む通常プレイの結果を必ずランキングへ残し、終了直後に自分の記録を見つけられるようにする。

### 実装内容

ランキングAPIに冒険ごとの `ranking_runs` テーブルを追加し、旧自己ベストを初回移行するSQLを作成した。通常プレイの終了時は自動送信し、送信後に週間ランキングを開く。APIが返す記録IDをクエリへ渡し、該当行だけを黄色の点滅で強調する。

### 設計上の注意点

テストモードは送信しない。ランキングは自己ベストではなく終了した冒険ごとの履歴であり、同じプレイヤーの複数行が表示される。

### 影響範囲

ランキング画面、Cloudflare Worker/D1、公開時のD1移行、ランキング仕様資料。
