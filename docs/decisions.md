# 設計判断（ADR）

## ADR-001 静的ゲーム本体と外部ランキングAPIを分離する

### 状況

ゲーム本体はGitHub Pagesで無料公開しているが、共有ランキングには永続DBとAPIが必要だった。

### 決定

ゲーム本体はGitHub Pagesの静的サイトのまま維持し、ランキングのみCloudflare WorkerとD1へ置く。

### 理由

既存の公開URLを変えず、無料枠で小規模ランキングを実現できるため。

### 却下した案

- GitHub Pagesだけでランキングを実装する: 書き込みDB/APIがない。
- ゲーム全体を別ホスティングへ移す: 公開経路を大きく変える必要がある。

### 今後の制約

Worker URLは `public/ranking-config.js` で管理する。ゲームの通常セーブをWorkerへ移さない。

## ADR-002 ランキングは任意送信のカジュアルランキングとする

### 状況

プレイ記録を共有したい一方、個人情報の自動送信やテストモードの混入を避ける必要がある。

### 決定

結果画面で本人が押した時だけ送信し、匿名端末IDごとの自己ベストを表示する。テストモード・途中放棄は除外する。

### 理由

プライバシーを保ちつつ、遊んだ結果を共有できるため。

### 却下した案

- 全プレイを自動送信: 意図しない外部送信になる。
- ログイン必須: 小規模ブラウザゲームの導入障壁が高い。

### 今後の制約

現在の順位は不正改変への完全な耐性を持たない。大会級の公正性が必要になった場合は、サーバー発行シードと行動ログ再計算を別ADRで採用する。

## ADR-003 リポジトリ内資料を長期開発の正式な記憶にする

### 状況

会話の圧縮・セッション切替だけでは、実装意図を継続できない。

### 決定

`AGENTS.md` と `docs/` を作業開始・完了時の基準にする。

### 理由

コードだけでは判断理由や未解決事項が残らず、次の作業者が安全に継続できないため。

### 今後の制約

機能、保存形式、外部連携、長期的な判断を変える変更は、同じコミットで資料も更新する。

## ADR-004 サイト訪問の計測はCloudflare Web Analyticsに限定する

### 状況

GitHub Pagesだけではゲームサイトの訪問数や流入元を把握できず、宣伝施策を比較できなかった。

### 決定

Cloudflare Web Analyticsのbeaconをページへ設置し、匿名のページ訪問集計だけを取得する。ゲーム行動イベントやセーブ内容の自動送信は今回追加しない。

### 理由

既存のCloudflareアカウントで無料に導入でき、GitHub Pagesの公開経路を変えずに日別訪問・流入元・端末傾向を見られるため。

### 却下した案

- ランキング送信件数をアクセス数とみなす: 任意送信のため実際の訪問数を表さない。
- ゲームプレイログをすべてWorkerへ送る: 既存の「端末内だけ」のテスト記録方針と衝突する。

### 今後の制約

計測対象を増やす場合は、送信項目・送信契機・画面上の説明を先に明記する。Cloudflareの計測トークンは公開用beacon設定であり、管理用APIトークンをリポジトリへ保存してはならない。
