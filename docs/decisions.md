# 設計判断（ADR）

## ADR-001 静的ゲーム本体と外部ランキングAPIを分離する

### 状況

ゲーム本体はGitHub Pagesで無料公開しているが、共有ランキングには永続DBとAPIが必要だった。

### 決定

ゲーム本体はGitHub Pagesの静的サイトのまま維持し、ランキングのみCloudflare WorkerとD1へ置く。

### 理由

既存の公開URLを変えず、無料枠で小規模ランキングを実現できるため。

### 却下した案

- GitHub Pagesだけでランキングを実装する: 書き込みDB/APIがない。
- ゲーム全体を別ホスティングへ移す: 公開経路を大きく変える必要がある。

### 今後の制約

Worker URLは `public/ranking-config.js` で管理する。ゲームの通常セーブをWorkerへ移さない。

## ADR-002 ランキングは任意送信のカジュアルランキングとする

> この判断の「任意送信」はADR-005で廃止し、通常プレイ終了時の自動送信へ変更した。カジュアルランキング、公開項目、テストモード除外の方針は継続する。

### 状況

プレイ記録を共有したい一方、個人情報の自動送信やテストモードの混入を避ける必要がある。

### 決定

結果画面で本人が押した時だけ送信し、匿名端末IDごとの自己ベストを表示する。テストモード・途中放棄は除外する。

### 理由

プライバシーを保ちつつ、遊んだ結果を共有できるため。

### 却下した案

- 全プレイを自動送信: 意図しない外部送信になる。
- ログイン必須: 小規模ブラウザゲームの導入障壁が高い。

### 今後の制約

現在の順位は不正改変への完全な耐性を持たない。大会級の公正性が必要になった場合は、サーバー発行シードと行動ログ再計算を別ADRで採用する。

## ADR-003 リポジトリ内資料を長期開発の正式な記憶にする

### 状況

会話の圧縮・セッション切替だけでは、実装意図を継続できない。

### 決定

`AGENTS.md` と `docs/` を作業開始・完了時の基準にする。

### 理由

コードだけでは判断理由や未解決事項が残らず、次の作業者が安全に継続できないため。

### 今後の制約

機能、保存形式、外部連携、長期的な判断を変える変更は、同じコミットで資料も更新する。

## ADR-004 サイト訪問の計測はCloudflare Web Analyticsに限定する

### 状況

GitHub Pagesだけではゲームサイトの訪問数や流入元を把握できず、宣伝施策を比較できなかった。

### 決定

Cloudflare Web Analyticsのbeaconをページへ設置し、匿名のページ訪問集計だけを取得する。ゲーム行動イベントやセーブ内容の自動送信は今回追加しない。

### 理由

既存のCloudflareアカウントで無料に導入でき、GitHub Pagesの公開経路を変えずに日別訪問・流入元・端末傾向を見られるため。

### 却下した案

- ランキング送信件数をアクセス数とみなす: 任意送信のため実際の訪問数を表さない。
- ゲームプレイログをすべてWorkerへ送る: 既存の「端末内だけ」のテスト記録方針と衝突する。

### 今後の制約

計測対象を増やす場合は、送信項目・送信契機・画面上の説明を先に明記する。Cloudflareの計測トークンは公開用beacon設定であり、管理用APIトークンをリポジトリへ保存してはならない。

## ADR-005 終了記録は自動送信し、ランキングでは冒険ごとに保持する

### 状況

任意送信かつ自己ベストだけの表示では、死亡や途中中断を含めた挑戦の記録が残らず、終了直後に自分の順位も確認できなかった。

### 決定

通常プレイの死亡・中断・帰還・踏破を自動送信し、D1へ冒険ごとの履歴として保存する。終了直後は週間ランキングを開き、今回作成された記録IDだけを点滅強調する。テストモードは送信しない。

### 理由

ランキングを挑戦の足跡として扱い、プレイヤーが追加操作なく今回の結果を確認できるため。

### 今後の制約

送信項目は従来の公開名、職業、到達階、得点、討伐数、階層主数、結果、日時に限定する。ランキングはカジュアル用途であり、クライアント改変への完全な対策は持たない。

## ADR-006 80点目標のプレイ時間・難易度・図鑑開示方針

### 状況

機能追加を続けるだけでは完成度の評価が上がりにくく、プレイ時間、難易度、収集情報の開示方針を固定する必要がある。

### 決定

- 100階踏破の標準所要時間を60〜90分とする。
- 難易度は「難しいが公平」を維持する。
- モンスターは発見式図鑑を維持する。
- アイテムは存在と分類を一覧表示し、詳細効果は入手後に解放する。
- 安定性、初見10分、攻略差、ビルド、バランス、演出の順で改善する。

### 理由

地下百景の緊張感と再挑戦性を残しつつ、理不尽さと情報不足を減らし、小規模作品としての統一された完成度を優先するため。

### 却下した案

- 45〜60分へ短縮する: 100階と装備ビルドの積み上げが薄くなる。
- 初見で容易に踏破できる難易度: 予告、防御、再挑戦の価値が弱くなる。
- 全攻略情報を最初から公開する: 発見と図鑑収集の動機を失う。

### 今後の制約

80点ロードマップ完了前は、無限階層、ガチャ、オンライン対戦、大量の追加職業など評価軸を広げる機能を優先しない。

## ADR-007 冒険セーブとランキング終了記録を通信・破損から復旧可能にする

### 状況

RunのJSON読込に失敗した場合は例外を無表示で捨てており、描画例外では黒画面になる可能性があった。またランキング送信が失敗すると終了記録を再送できなかった。

### 決定

- Runを書き換える前に直前の正常値を1世代バックアップする。
- 読み込めない現在値は削除せず隔離し、正常なバックアップを自動復元する。
- 描画例外では復旧画面を表示する。
- ランキング終了記録は送信前に端末キューへ保存し、成功後に削除する。
- 冒険ごとの `submissionId` をD1で一意にし、同じ記録の再送を冪等にする。

### 理由

ブラウザ終了、通信断、公開更新中の不整合が起きても、冒険と終了記録を失わず、プレイヤー自身が復旧できるようにするため。

### 却下した案

- 読込失敗時に自動で全セーブを削除する: 復旧可能なデータまで失う。
- ランキング送信を終了画面から手動でやり直す: 強制記録方針と合わず、閉じると失われる。
- 再送のたびに新規行を作る: 通信応答だけ失われた場合に二重登録になる。

### 今後の制約

正常終了では現在Runとバックアップを削除する。隔離データは自動削除しない。ランキングの送信項目を増やす場合も `submissionId` は維持し、旧公開版との切替期間はWorker側の後方互換を確保する。

## ADR-008 英語対応は第2.5段階で基盤、第6.5段階で全面対応する

> 基盤実装の時期はADR-010の敵攻略文言を先に確定するため、第3段階直後へ順序変更した。第6.5段階で全面対応する方針は継続する。

### 状況

海外からの集客には英語対応が有効だが、現状の画面文言はゲーム本体へ直書きされており、安定性と初見導線より先に全面翻訳すると修正量と回帰範囲が大きい。

### 決定

第3段階の日本語戦闘・図鑑文言確定後、第2.5段階で翻訳キー・言語非依存ID・切替機構を整える。絵・音・演出が固まった後の第6.5段階で英訳、英語レイアウト、英語話者テストを完了する。

### 理由

翻訳のやり直しを抑えながら、後半に一括移植するより早い段階から言語依存を除去できるため。

### 今後の制約

セーブ、ランキング、マスターデータの識別子へ表示文言を保存しない。翻訳漏れは日本語へフォールバックし、言語設定で進行不能を起こさない。

## ADR-009 ランキングは利用者が増えるまで総合順位へ一本化する

### 状況

週間と総合に記録を分けると、現在の参加人数では週間側が空または少人数になり、順位表の賑わいと継続性が弱く見える。

### 決定

ゲーム画面とランキング取得APIは総合順位だけを表示する。タイトル画面も総合上位3件にする。既存記録とD1の `week_key` は削除せず、利用者が増えた際に週間表示を再開できる状態を保つ。

### 理由

挑戦記録を一つの順位表へ集め、少人数でも順位が積み上がる価値を出すため。

### 却下した案

- 週間だけを残す: 週替わりで少ない記録が消え、順位表が空に戻りやすい。
- 週間と総合を現状のまま併設する: 少ない記録を二つへ分散する。
- 週間データ列を削除する: 将来の再開に不要な移行作業が発生する。

### 今後の制約

週間再開は十分な週間投稿数を確認してから行う。それまではUIへ週間タブや週間TOP3を表示しない。

## ADR-010 敵の攻略差は1〜20階の縦切りから全階へ広げる

> 2026-07-23に25素体・4変異と全10階層主へ展開した。図鑑IDは既存100件を維持し、階層主を図鑑100枠へ含める再編だけは移行設計まで保留する。

### 状況

100体の名前・弱点・図鑑は存在するが、戦闘行動は共通乱数が中心で、弱点も実際のダメージへ接続されていなかった。100体を一度に変更すると難易度変化と不具合の原因を特定しにくい。

### 決定

最初に1〜20階の10素体と10・20階の階層主を完成形として実装する。通常敵は固有行動、予告、弱点、耐性、攻略法を共通マスターへ持ち、図鑑情報は遭遇・1・3・5体討伐で段階開示する。固有能力の追加分はHP・攻撃倍率を下げ、平均脅威を旧基準以下に保つ。

### 理由

短い区間で全職業・スマートフォン表示・セーブ互換・攻略理解を確認してから、同じ構造を残りへ安全に展開できるため。

### 却下した案

- 100体を同時に実装する：調整対象が多く、到達率低下の原因を追えない。
- 固有能力を追加して数値を据え置く：判断要素ではなく単純な難化になる。
- 弱点を図鑑表示だけに残す：攻略情報と実際の戦闘が一致しない。

### 今後の制約

敵の強行動には画面上の兆候か次行動予告を必ず付ける。固有能力追加時は旧基準との脅威比較を行う。既存図鑑IDを変更する場合は、階層主10体を含む100体への移行と旧討伐数の保持方法を先に設計する。

## ADR-011 深層の敵番号は100加算で新旧を判別する

### 状況

従来の `DungeonEnemy.kind` は素体番号として保存され、図鑑の4変異を一意に表せなかった。意味をそのまま図鑑添字へ変えると、冒険中の旧セーブが別の敵へ化ける。

### 決定

新しく21階以降へ生成する通常敵は「図鑑添字 + 100」を `kind` に保存する。100以上は新形式、100未満は旧形式として読み分ける。旧戦闘に `catalogId` がない場合も旧形式から補完する。

### 理由

Runのバージョン変更や全敵オブジェクトの移行をせず、旧セーブを保持しながら4変異を実戦へ出せるため。

### 却下した案

- `kind` を図鑑添字へ即時置換する：旧Runの番号と衝突する。
- Run v4へ一括移行する：今回変更しない地図・装備まで移行対象が広がる。
- 表示名だけを変異化する：行動、弱点、図鑑記録が同一のままになる。

### 今後の制約

100未満を旧形式として読む互換分岐を削除しない。敵識別子を再変更する場合は、探索中の敵と戦闘中・反撃待ちの敵を両方移行する。
