# 地下百景 開発ガイド

このリポジトリ内のコードと資料が、地下百景の正式な記憶である。会話履歴だけを根拠に実装しない。

## 作業開始時

1. `README.md` と `docs/now.md` を読む。
2. 変更範囲に対応する資料を読む（戦闘なら `gameplay.md`、保存・ランキングなら `data-and-save.md`、構成変更なら `architecture.md`）。
3. 関連コードの呼び出し元・呼び出し先を確認する。
4. 過去の意図が必要なら `docs/decisions.md` と `docs/implementation-log.md` を参照する。

資料とコードが食い違う場合、通常は現在動いているコードを事実として扱い、資料を更新候補として報告する。セーブデータ、外部送信、課金、公開設定に関わる食い違いは、推測で変更しない。

## 情報の優先順位

1. 現在実行されているコードとテスト
2. `docs/data-and-save.md`
3. `docs/decisions.md`
4. `docs/architecture.md` と `docs/gameplay.md`
5. `docs/implementation-log.md`
6. `docs/todo.md`

## 実装方針

- 既存の状態管理・命名・責務分担に合わせ、必要最小限を変更する。
- 機能追加と大規模リファクタリングは原則分ける。
- 既存セーブを破壊する変更、外部へデータを送る変更、公開先の変更は、影響・移行・ロールバックを事前に示す。
- 一時対応を恒久化しない。既存処理を重複して作らない。
- 無関係な整形、ファイル移動、命名変更を混ぜない。

## ドキュメント更新

機能、仕様、保存形式、外部連携、設計判断を変えたら、関連資料をコードと同じ変更に含める。

- 小さな表示・文言修正: `docs/now.md` または履歴更新は不要。
- 機能追加・不具合修正: `docs/implementation-log.md` を更新。
- 今後も拘束する選択: `docs/decisions.md` にADRを追加。
- 未解決事項: `docs/todo.md` を更新。

## 完了前の確認

1. 変更差分を読み直す。
2. 関連する既存機能の回帰を確認する。
3. 可能な範囲で `npm test` と `npm run build:pages` を実行する。
4. 実行できなかった確認は、完了報告と履歴に「未確認」と記す。
5. 資料と実装の整合を確認する。

## 完了報告

簡潔に、実装内容・変更ファイル・確認結果・資料更新・未対応事項を報告する。
